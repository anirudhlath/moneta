# moneta

Personal finance app answering one question: **monthly spending power = income − fixed costs**. FastAPI server owns all logic; the typer CLI is a thin HTTP client (in-process ASGI when no server is running). Design doc: `docs/superpowers/specs/2026-07-07-moneta-design.md`.

## Verification

```bash
uv run pytest -q && uv run ruff check . && uv run ruff format . && uv run mypy --strict src tests
```

All four must pass before every commit. Test output must be pristine — a deprecation warning is a failure.

Run it: `uv run moneta --help` (CLI, no server needed — in-process ASGI) or `uv run moneta serve` (real server; see README for the full quickstart).

## Conventions that aren't obvious from the code

- **Money is integer cents everywhere** (`*_cents: int`) — including every API response field; `Decimal` only at the aggregator input boundary via `to_cents` (models.py). `dollars()` renders unsigned dollars for prose only (LLM prompts, review-question text); the CLI formats every money cell via `fmt_money` (`-$X.YY` negatives) or, for a field documented as an unsigned magnitude, `fmt_outflow` (`fmt_money(-magnitude_cents)`). Never float for money. Share quantities on `Holding` are float — shares aren't money. API money fields split by field, not globally: flow fields are signed (storage convention), aggregate/labeled-quantity fields are unsigned magnitudes — see the per-field sign table in the user guide's server-mode section.
- **Sign convention:** negative = outflow, positive = inflow (SimpleFIN's convention, kept end-to-end).
- **Plaid inverts both signs** (aggregator/plaid.py): Plaid amounts are positive when money leaves the account and liability balances are positive-owed; the adapter negates both so stored data follows the SimpleFIN convention. `PlaidAdapter.fetch` ignores `since` — it replays `/transactions/sync` from an empty cursor every run (≤730 days; ingest dedup absorbs the overlap), so `sync --full` is a no-op for Plaid.
- **Enum columns load as plain `str`**, not enum instances (columns are `String`-typed). Compare with `==` (StrEnum equals its value); never `is`, never `.name` on loaded values.
- **Pipelines commit; views don't.** `pipelines/*` own their transaction boundary (`session.commit()` inside); `views/*` are pure reads. `vesting.apply_vesting` commits (it's an import pipeline).
- **Pipeline order is load-bearing:** `run_sync(session, adapters: list[AggregatorAdapter], llm, today, full)` = fetch (per-adapter) → ingest → normalize → transfers → financing-detect → auto-review (LLM, when configured) → recurring → verify → events. Financing-detect (`pipelines/financing.py`) reads only transfers' account data, so it sits right after transfers, keeping every question-opening stage adjacent and before auto-review. Auto-review runs before detection so confident LLM answers to open review items shape this run's series/exclusions; recurring detection reads transfer links; `verify_series` second-opinions what detection produced before events fire on it; events read series. `detect_recurring` must never rewind `next_expected_on` (it takes `max`) — rewinding re-fires missed events on every sync. `run_sync` brackets the stages with a `SyncRun` audit row (success + report, or the error, surfaced by `moneta status`). The fetch step (`_fetch_all`) iterates `adapters` itself — no merging layer — collecting each adapter's `Snapshot.warnings` (and, per adapter, a synthesized `"{source}: {ExcType}: {exc}"` warning on fetch failure) into `SyncReport.warnings`; a failing adapter is skipped (warn + continue) *unless it's the only adapter or every adapter fails*, in which case the first failure re-raises and the whole sync fails — an empty snapshot must never be ingested as a quiet success, since `data_as_of`/staleness would then lie about having fresh data.
- **Schema changes need an Alembic revision** in `src/moneta/migrations/versions/` (`NNNN_name.py`, `down_revision` chained to the previous) **plus a `_HEAD` bump in `db.py`** — `init_db` fast-paths past all alembic machinery when `alembic_version == _HEAD`. `init_db` runs `upgrade head` and adopts a pre-Alembic DB by stamping the `0001` baseline — never call `create_all` in production code. `tests/test_migrations.py` pins migration/`Base.metadata` parity and `_HEAD` against the script directory, so forgetting either fails the suite.
- **Events semantics:** `emit_series_events` catches up every missed period in one sync (one `missed` event per empty grace window), and a `price_increase` needs the two newest occurrences to agree (>5% drift from expected, within 5% of each other) — one outlier must never rewrite `expected_cents`. When a resumed series makes detection leap `next_expected_on`, `detect_recurring` backfills misses for the skipped windows itself, bounded at the newest charge (windows past it belong to events' future-guarded loop).
- **Dates are local-timezone:** `_ts_to_date` converts SimpleFIN timestamps in the process's local tz (the user's calendar day). Tests pin `TZ=UTC` via an autouse conftest fixture; date-sensitive tests set their own TZ.
- **One currency end-to-end:** `queries.primary_currency` picks the majority currency (ties prefer USD). networth/power/cashflow filter their sums to it (net worth reports `foreign_accounts` excluded), and `detect_recurring` only groups primary-currency transactions — so series, and the income/fixed-cost numbers derived from them, are single-currency by construction.
- **Sync window is per-source, not global:** `_sync_since(session, full, source)` computes each adapter's `since` independently, from the newest `posted_on` of transactions whose `Account.source == source` (`Account.source`, migration `0004`, stamped by every `AccountDTO`/adapter). First sync and `sync --full` request from the epoch (1970-01-01) for every source — there is deliberately no window constant; a normal re-sync resumes each source from *that source's own* newest stored txn minus a 7-day overlap (`RESYNC_OVERLAP_DAYS`, `pipelines/run.py`). This is the fix for the old global-window bug: Plaid's daily full-history replay could no longer mask a stalled SimpleFIN source, because SimpleFIN's window is derived only from SimpleFIN-sourced transactions. Fallback: when a source has no accounts yet (first run after adopting `Account.source`, or a genuinely new source's very first sync on an already-populated DB), its `since` falls back to the *global* newest txn rather than the epoch — deliberate/accepted, so use `sync --full` after linking a brand-new source to avoid a truncated first pull. The SimpleFIN adapter satisfies a deep `since` by walking ≤45-day windows backward (the beta bridge hard-caps any request to the trailing 90 days of the range) and stops after ~1 year of empty windows; that's transport policy and lives in the adapter, not the pipelines. There is no `MergedAdapter` — `run_sync` takes `adapters: list[AggregatorAdapter]` and iterates them itself (each with its own `since`, merged into one `Snapshot` before a single `ingest_snapshot` call); `api._build_adapters(settings) -> list[...]` builds the list, and `POST /sync` 400s when it's empty.
- **Series lifecycle:** `detect_recurring` auto-ends a series whose newest occurrence is >3 cadence periods before `today` (ended series are skipped by events and the power view). An ended series — auto or manual — reactivates only when the group's newest significant txn is untagged (`series_id IS NULL`, i.e. genuinely new since the last run) and fresh; backfilled old txns never reactivate. Upstream description corrections keep their series tag (ingest clears only `merchant`; detection untags only if the re-derived merchant actually moved) so a corrected old txn can't fake a revival; manual reactivation via PATCH bumps `next_expected_on` forward to today. Cadence and amount stats are judged on the newest run of occurrences, not the whole group, so breaks in deep history don't poison detection.
- **Discretionary semantics:** a series can be `discretionary` (a habit, not a bill) — set by an amount-unstable group's LLM "habit" answer, or a human's `--habit`/`recurring_cluster` b/h/n resolution (`--re-review` only reopens the question; the follow-up answer sets the flag). Discretionary series are still fully tracked (active/ended lifecycle, cadence, price events) but `views/power.py` excludes them from `fixed_costs` unconditionally and instead counts their transactions in `spent_so_far` (the same bucket as untagged and ended-series txns). The flag lives in the `recurring_cluster` force map (`detect_recurring`'s `force: dict[(merchant, direction) -> (is_recurring, discretionary)]`), so it's re-applied and persisted onto the series (`series.discretionary = discretionary`) every sync — a resolved habit answer never has to be re-asked.
- **LLM boundary (design §9):** every pipeline takes `llm: Classifier | None`. LLM output is classification only — it gates decisions, never supplies a money value; ambiguity degrades to a `ReviewItem` when no LLM is configured. Keep it that way.
- **LLM verification is a ledger, not a veto:** `verify_series` reviews each active series once — a `recurring_cluster` ReviewItem (open or resolved), keyed by `series_key(merchant, direction)` (models.py), is the per-series record. Verification is three-way (`bill`/`habit`/`not_recurring`, same as detection's own classification): confident `bill` → resolved item (`is_recurring: true`, feeds detection's force map); anything else (confident `habit`/`not_recurring`, unconfident, or malformed) → open item with `payload.llm_flagged: true` and `payload.llm_leaning` set to the parsed classification (or `"unparseable"`) so a human reviewing it sees what the LLM thought without it deciding anything. Flagged items are human-only (`autoreview_items` skips them), and the series stays active until a human rules via `moneta review`'s b/h/n prompt — the LLM never suppresses a deterministic detection. Price drift >5% in events is LLM-gated the same way: not confident-yes → `price_change` item; resolving it true applies the txn's amount (with no LLM configured, both paths degrade to the deterministic rules — detections stand as-is and a price change applies only via the two-occurrence agreement rule — and open nothing).
- **Transfer-link semantics:** both legs of every transfer link are unconditionally excluded from recurring-merchant grouping and from `spent_so_far` (`queries.linked_txn_ids`) — a payment to a loan account no longer forms (or feeds) a merchant series at all. Loan-like payments (account `type == loan` or `financing_mode`) are instead derived **per-account** at view time from the linked outflows via `queries.loan_payment_stats` (cadence + amount stats grouped by inflow account, so a bank collapsing several cards' payments into one shared descriptor can't blend or drop an obligation), and surfaced as fixed-cost lines in `power` / rows in `obligations` directly — not via detection. Credit-payment series formed before this shipped are still filtered out of fixed costs in the power view (their purchases are counted instead, no double count). Shared link classification lives in `queries.classified_links` — extend it rather than re-joining TransferLink at call sites.

## Layout

- `aggregator/` — adapter protocol (`AggregatorAdapter`: `source: str` property + `fetch`) + SimpleFIN + Plaid; `run_sync` iterates the configured adapters itself, no merging layer; DTOs stop at `pipelines/ingest.py`
- `cadence.py` — pure cadence math (gap matching, monthlyizing, tolerances); no I/O, shared by `queries.py` and `pipelines/recurring.py`
- `pipelines/` — ingest, normalize, transfers, financing (deterministic financing-mode fingerprint + one-time review question), review (LLM auto-review + `verify_series`), recurring, events, run (orchestrator)
- `views/` — power (flagship), cashflow, networth, financing (obligations are derived, not stored)
- `api.py` — all endpoints (optional bearer-token auth via `api_token`); `cli/` — thin client, zero business logic
- `queries.py` — shared cross-table lookups (`classified_links`, `account_type_map`, `primary_currency`)
- `migrations/` — Alembic env + hand-written revisions, run programmatically by `db.init_db`
- `logs.py` — loguru sinks (stderr warnings + rotating `moneta.log`), configured per config dir in `build_app`

## Gotchas

- `httpx.ASGITransport` never fires FastAPI lifespan — the CLI's in-process path drives the app's own lifespan manually around each request (`app.router.lifespan_context`, cli/client.py), so `init_db` and engine disposal happen exactly where a real `moneta serve` would run them.
- Endpoints resolve `date.today()` at request time; test data must be date-relative (see tests/test_e2e.py's anchoring comment).
- SimpleFIN gives no account types — they're inferred from name/org keywords (`pipelines/ingest.py`); Plaid supplies real types via `AccountDTO.type_hint`, which beats inference; user overrides via `moneta accounts --set-type` survive re-sync.
- Backlog convention: `docs/backlog/<priority>/<kebab-case>.md`; QA items in `docs/qa-backlog/`.
